I am working on a project to make an open-source browser based Flash Authoring system.
It is a HyperText Markup Language and Java Script based Shockwave Flash Animator, using pre-existant client-side ajax SVG editors with modified runtimes to create alternative scripts. Since XML, SVG, and HTML technologies are not yet powerful to create actual swf files, it will use a Java Applet for SWF bytecode compiliation through JavaScript aggregated scripts using open source classes and libraries on Java Enabled Platforms. If that sort of platform you are using is not Java Enabled, an alternative could exist by uploading the script via HTTP Post to a server capable of natively, or through CGI, generating the SWF file throught (almost) the same process of the Java Applet, and sending it back to the HTML renderer, or Flash IDE container

Note:

IE might not be supported because of how it only supports VML, unless I manage to find a IE hack that emulates SVG on IE

It's not ajax because it's client side, other than the uploading of the Scripts to a server for non Java-Enabled Platforms to generate SWF files.

Goal:

Creating an open source browser-based IDE for flash development.

Process of developing:

Finding an open-source SVG Editor
Make a partial SVG to SWFML? translator
Making a timeline system
Making a script to combine SWFML? parts
Generating the swf through a java applet


Proposed Functionality in early releases:
Completely Client-Side
Platform Independant
Completely Open Source
HTML/JS/SVG Frontend
JS/Java Backend
SAAS components (Software As A Service) (for non-java-enabled platforms)
Drawing Lines
Drawing Freeform
Drawing Squares
Drawing Eclipses
Java Applet for SWF bytecode compiliation through JavaScript aggregated scripts using open source classes and libraries

F.A.Q.
Q. Why don't you just code the whole thing in java?
A. HTML/JS is much more platform Independent than Java. The idea is to make an animation on the site, save to the site's server, and continue working on it later. Later being able to publish it and be sort of a YouTube for Flash Animations/Applications. Another aim of this project is to be next-gen software, sure there are about 10 open source multiplatform C++ based flash IDEs out there, but none of them truely represend the future of software, or true platform independence.
Q. Is there going to be ActionScript support?
A. Probably No. I don't wan't this to be fully fledged, as most of the people who will probably be using this will be kids who just want to amaze their friends by making animations.
Q. Why don't you have more questions in the F.A.Q.?
A. I can't think on any other questions

Adobe practically has a monopoly on the Flash market; ask any Flash animator what version they use and chances are it is an "official" Macromedia/Adobe version, despite the presence of many different (and cheaper) alternatives; with Liveswif being the only freeware Flash program for Windows I know of.

A quality open source replacement, promoted well via word of mouth, could improve things for consumers in the end by providing them with a choice, increasing the need for competition, causing the other competitors to drop their prices or add more "exclusive" features. A quality web-based Flash authoring tool could also introduce more people into the established userbase by providing a friendly, convenient and most importantly, free/cheap method of making animations/slideshows and related projects.

In order to even be considered as a "quality" replacement by the majority of the Flash animation userbase, a Flash program would have to comply with the latest common plugin standard, and run smoothly. Sadly, Liveswif has quite a few annoying bugs, and only runs on Flash 5.

I'd like to see this browser-based Flash program be successful. It provides the convenience of a browser-based application, yet delivers the functionality of a basic Flash program, and could well be the first of its kind.

This project is indeed ambitious; don't give up, no matter what. If you keep working on the project, and keep spreading this project by word of mouth throughout the Flash community, whether it is by discussing it on other Flash forums or other forums in general, it will be worth the effort.

Great work so far, and good luck in the future.

Also, good software engineering dictates that all right-click actions should also be available via a left click menu. On Firefox, it is possible to disallow all webpages to disable the suppression of the context menu (basically, the menu you made is covered up by the browser context menu). This does not mean that the context menus are not useful, but that menu should not be the only way to access any given function.
