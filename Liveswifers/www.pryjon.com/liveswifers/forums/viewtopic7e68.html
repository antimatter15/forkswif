Yes I'm familiar with Boomerang, though I thought they stopped production two years ago.

The problem with trying to convert ASM into C++ comes in the two following areas.

First there's object oriented programming. Many objects in C++ are not defined in ASM, in fact usually when you create a class and use it the process to convert it into machine code takes most of the structure away.

Also different compilers, different compiling options, not to mention different object modes produce different forms of machine code at the end of the compilation. Here are two examples of the same thing in ASM

mov ax, 1
mov bx, 2
add ax,bx

That adds 1 and 2 and puts the result in ax

mov ax, 1
add ax, 2

That does the same thing but the machine code is different. So a decompiler or converter would have to recognize this difference and make the choice accordingly. Now adding 2 to a number isn't difficult, however this is just an example of how the same process can be done in two completely different ways.

Another thing to consider is that in C++ all loops are based upon the while loop. This shouldn't be a problem as for(int ii, ii < 5, ii++) should act the same whether it's in the form of a for loop or a while loop.

int ii
while( ii < 5)
{
ii++
}

Then of course there's the fact that many variables that had significant names are lost, so only vague references exist.

None of this makes learning the code impossible, however it makes it very difficult and time consuming. Not to mention you have to keep track of the windows API calls inside of Assembly. From what I've seen they only use the basics, but this GEO.dll has me stumped. I need to find out what this dll is before I can properly understand its use.

It was C, technically since C++ is simply a revamped version of C designed for Object Oriented Programming you could call the general code created by RECStudio either C or C++. But the site says it's meant to be C.

You know, I had entirely forgotten about REC. I recently redownloaded and was pleasantly surprised to find they'd added that wonderful GUI. Definately beats the old method. The author of the REC decompiler was the one whom originally pointed me towards Boomerang.

It was designed to convert disassembled code into C code. However since much of the original data is lost in compilation all it does is surround ASM code with C code.

Basically, in truth, REC doesn't completely decompile to C. You see, Boomerang uses more advanced algorithms to help determine variables, functions, and so on on a better scale than REC.

The author of REC has even said so himself.

However most of this general information is lost when compiling code, and it's worse when they use Optomised code generation at compile time.

In stead of giving variable names, memory addresses are calculated to perform this task.

So even this won't help, much that is. Basically I used to love REC because it removed much of the redundancy of trying to match each calculation movement. It will interrpret while loops, for loops, if thens, case statemetns and so on. Even a function or two hundred, but in the end the rest is up to the user to decrypt the ASM. So though easier, still going the ASM route to finish up the rest.
